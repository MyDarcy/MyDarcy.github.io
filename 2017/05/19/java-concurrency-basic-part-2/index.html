Java并发基础-part-2

volatile与Java内存模型

  • 为了保证线程之间的可见性，有序性，原子性;Java使用了一些特殊的指令或者关键字告诉虚拟机在这些地方不要随意优化目标指令，如volatile。
  • volatile: 易失的, 易变的
  • volatile并不能代替锁，不能保证操作的原子性质(对于那种plus任务，使用AtomicInteger就可以了);
  • volatile能够保证数据的可见行和有序性；
  • 虚拟机的Client模式下，JIT并没有做足够的优化，主线程修改ready变量的状态，ReaderThread会发现这个改变，并且退出，但是如果在Server模式下，系统优化后，ReaderThread将会无法看到主线程的修改，导致ReaderThread永远不会退出。
  • 这里只需要通过volatile来声明ready变量，那么主线程对ready做的修改在其他线程就是可见的了。

线程组

  • 相同功能的线程放到一个线程组中；

守护线程

  • 守护线程是一类特殊的线程，GC线程，JIT线程都是守护线程；与之对应的是用户线程，如果用户线程全部结束，那么守护线程要守护的对象也就不存在了，所以当一个Java应用中只有守护线程的时候，Java虚拟机退出；

线程优先级

  • Java线程的优先级和底层的操作系统有关系；无法精确控制；

demo

synchronized关键字

  • volatileb不能保证原子性(多条指令的执行不能保证原子性)，之前的例子也有提到（PlusTask）。
  • 关键字synchronized是对同步的代码块进行加锁，使得每次只有一个线程进入同步块，从而保证线程的安全性。
  • 关键字synchronized的用法
    • 指定加锁对象: 对给定对象加锁，进入同步代码块之前要先获得给定对象的锁
    • 直接作用于实例方法： 相当于对当前实例加锁，进入同步代码块之前要先获得当前实例的锁
    • 直接作用于静态方法： 相当于对当前类加锁，进入同步代码块之前要先获得当前类的锁

错误的用法

  • 两个线程指向了不同的Runnable对象， 两个线程使用不同的锁对象，无法保证线程安全。使用static synchronized 修饰increase方法就能保证正确。
  • why ??? — 因为即使线程使用不同的Runnable实例, 但是方法块请求获取的锁对象是当前类的锁。而不是到那个当前实例，所以仍然可以保证正确。

还有

  • synchronized还可以保证线程间的可见性和有序性。
  • synchronized限制任何时候都只有一个线程访问可以访问同步块，因此只要保证串行语义一致，执行的结果都是一致的。即被synchronized限制的多个线程是串行执行的。

并发下的ArrayList

  • 可能的问题
    • ArrayList扩容的过程中，缺少锁的保护，内部一致性被破坏，但是由于没有锁的保护，另外一个线程访问到了不一致的内部状态，导致出现了越界错误。
    • 多个线程访问到同一个位置，覆盖了， 输出数字小于200000。

并发下的HashMap

  • 可能的问题;
    • 结果不是预期值
    • 无法退出循环: 扩容时产生了环结构。(Java8中已经不会出现这样的错了，但是仍然会出现数据不一致的情况)
  • 解决方案：ConcurrentHashMap

错误的加锁

  • 程序的输出不是200000,而是比200000小的数字
  • 把锁加在Integer变量i上产生的问题；因为会不断的通过Integer.valueOf()创建新的对象，这样多个线程加锁都不在一个对象上就产生了这样的问题；
  • 解决方案: synchronized(instance)
