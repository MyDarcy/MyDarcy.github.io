主要介绍CompletableFuture, StampedLock, 增强的原子类LongAdder和LongAccumulator

CompletableFuture: 增强的Future

  • CompletableFuture实现了Future接口，同时也实现了CompletionStage接口，CompletionStage 接口拥有多达40种方法，是为了函数式编程中的流式调用准备的。通过CompletionStage提供的接口，就可以在一个执行结果上进行多次流式调用。

CompletableFuture

  • CompletableFuture和Future一样，可以作为函数调用的契约，如果向CompletableFuture请求数据而数据没有准备好，那么当前线程就会阻塞等待。可以手动设置CompletableFuture的完成状态。
  • supplyAsync()方法用于需要有返回值的场景(特定的计算)，而runAsync()方法用于没有返回值的场景(只是执行异步的工作)。重载版本可以接收一个executor参数，让这个任务在指定的线程池中执行。而如果不指定的话，则是在默认的公共的ForkJoinPool.common线程池中执行。
  • Java8中新增ForkJoinPool.commonPool()方法，它可以获得一个公共的ForkJoin线程池，这个公共线程池中的线程都是Deaemon线程，所以如果主线程退出，那么整个系统就会退出而不管这些Daemon线程执行完毕没有。

读写锁的改进： StampedLock

  • 读写锁分离了读和写的功能，读读之间完全并发，但是读写，写写是阻塞的。使用的依然是悲观的锁策略。如果右大量的读线程，那么依然会引起写线程的阻塞。
  • StampedLock提供了乐观的读策略。类似于无锁操作。

StampedLock

  • StampedLock内部实现使用类似于CAS操作的死循环反复尝试。挂起线程使用的是Unsafe.park函数，park函数在遇到线程中断时，会直接返回。而在StampedLock的死循环逻辑中，没有处理有关中断的逻辑，所以，这会导致阻塞在park()上的线程被中断后，会再次进入循环。而当退出条件得不到满足时，疯狂占用CPU。

StampedLock的实现思想

  • StampedLock的内部实现是基于CLH锁的，CLH锁是一种自旋锁。保证FIFO，保证没有饥饿的发送。
  • CLH锁的基本思想：维护一个等待线程的队列，所有申请锁但是没有成功的线程都记录在这个队列中。每一个节点(代表一个线程)， 保存了一个标记位(locked), 用于判断当前线程是否释放锁。只要前序节点没有释放锁，则表示当前线程不能继续执行，所以会自旋等待。而当锁释放时，线程会将自身的节点的locked标记为false， 这样后续的等待线程就能继续执行了。

原子类的增强

LongAdder

  • AtomicInteger的基本实现机制：在一个死循环中，不断尝试修改目标值，直到修改成功。如果竞争不激烈，那么修改成功的概率很高，否则，修改失败的概率很高。大量修改失败时，多次循环尝试，性能收到影响。
  • 竞争激烈时，如果进一步提高系统性能
    • 热点分离，将竞争的数据进行分解。可行的方案是仿造JDK1.7中ConcurrentHashMap的实现，将热点数据进行分离（成为数组）。实际的操作中，LongAdder并不是一开始就动用数组进行处理。而是将所有数据都记录在一个称为base的变量中。多线程环境中，如果base修改没有冲突，那么就不需要动用cell数组；而一旦base修改冲突，就会初始化cell数组。如果使用拓展cell更新后，某一个cell的更新仍然发送冲突，那么系统就尝试创建新的cell或者扩容，以减少冲突。
  • 从实验效果来看，Synchronized, Atomic, LongAdder的性能略大，LongAdder的性能反而差一些，Atomic性能最好。
